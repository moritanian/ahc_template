# 焼きなまし (Simulated Annealing) 実装ガイド (AHC)

---

## 1. 基本骨格

`lib/my_lib.cpp` の `TimeKeeper` を使った典型的な実装:

```cpp
// 温度パラメータ (問題ごとに実測チューニング)
const double T_start = 1e4;
const double T_end   = 1e1;

mt19937_64 rng(42);

auto best_sol = current_sol;
int  best_score = current_score;
int  iter = 0;

while (!time_keeper.is_time_over()) {
    ++iter;
    double ratio = time_keeper.get_elapsed_ratio();           // 0.0 → 1.0
    double temp  = T_start * pow(T_end / T_start, ratio);    // 指数冷却

    // 近傍を生成して差分スコアを計算
    auto [action, delta] = generate_neighbor(rng);

    if (delta > 0 || rng() < (uint64_t)(exp((double)delta / temp) * (double)UINT64_MAX)) {
        apply(action);          // 採択
        current_score += delta;
        if (current_score > best_score) {
            best_score = current_score;
            best_sol   = current_sol;
        }
    } else {
        revert(action);         // 棄却
    }
}
// best_sol を出力
```

**スコアが最小化の場合**: `delta < 0` で採択、遷移確率は `exp(-delta / temp)`

---

## 2. 温度スケジュール

### 指数冷却 (推奨)

```cpp
double temp = T_start * pow(T_end / T_start, ratio);
```

- `ratio = time_keeper.get_elapsed_ratio()` で自動的に時間に比例
- **T_start / T_end の比** が探索の広さを決める。比が大きいほど序盤に広く探索
- チューニング方針: 採択率を序盤 80〜90%、終盤 1〜5% 程度に調整

### 採択率の確認方法

```cpp
int accept_cnt = 0, reject_cnt = 0;
// ループ内で採択時に ++accept_cnt, 棄却時に ++reject_cnt
cerr << "accept rate: " << (double)accept_cnt / (accept_cnt + reject_cnt) << endl;
```

## 3. 初期解の設計 (重要)

**AHC058 公式解説**: 複雑な初期解より **シンプルな初期解** が良い場合が多い。
- 長い初期解を作ると「投資先を変える」ような大きな変更が受け入れられにくくなる
- 強い初期解が必要なのは「谷が深い」問題 (チェックポイント方式 → 後述)

**AHC051 公式解説**: スコアが **積** で表される問題は初期解に特に注意。
- 悪い初期解では小さい要素の改善が全体スコアにほぼ寄与しない (`1e-10 → 1e-9` でも無意味)
- 対策: 全要素が繋がった初期解を作る、または相対スコアで評価する

## 4. 近傍設計

### 基本パターン

| 近傍の種類 | 適する問題 |
|---|---|
| 1点変更 | 割り当て・配置問題 |
| swap | 順列・順序問題 |
| 2-opt / 3-opt | 経路問題 |
| 追加・削除 | 集合選択問題 |
| 区間反転 | 順序問題 |

### 近傍設計のコツ

- **解の構造を観察してから近傍を設計する**: 良い解を出力して傾向を確認し、それを活かす近傍を追加 (AHC051)
- **候補を絞って近傍を重み付き選択**: 全候補でなく上位 X 件に絞る、良い候補が選ばれやすい重みをつける (AHC055)
- **回数が回らない場合は複数まとめて変更**: 近傍 1 回の変更量を増やす (AHC055)

### 順序問題の特殊テクニック (AHC055 公式解説)

順序を直接 encode するのではなく、**要素ごとの優先度 (重み) を焼く**:

```cpp
// 悪い例: 順列そのものを近傍 (swap) で焼く
vector<int> order = {0, 1, 2, ...};

// 良い例: 優先度を焼いて順序は都度ソートで決定
vector<double> priority(N, 1.0);
// 近傍: priority[i] を微小変化
// 順序決定: sort by priority
```

これにより 1 点変更で実質的に「任意の位置への挿入」相当の探索ができる。

## 5. 評価関数の設計

### 中間状態の評価 (AHC058 公式解説)

最終スコアだけでは中間状態の良し悪しが判断できない場合、**各ターンの評価を加算**する:

```cpp
// 例: 各ターンの対数値を加算して中間状態を評価
double score = final_score + lambda * sum_of_log_per_turn;
```

- 終盤に指数的にスコアが増える問題で特に有効
- λ は実測でチューニング

### 「対価を受け取るタイミング」にボーナス (AHC055 公式解説)

均等に評価するのでなく、報酬が得られるタイミングの遷移に bonus を与える:

```cpp
// 例: 宝箱を開封できる行動に +bonus
if (can_open_chest) score += open_bonus;
```

### スコアが積の場合

- log に変換して加算に変換すると差分計算が安定
- 要素が 0 に近い場合は小さな改善が全体に影響しない → 相対スコアで評価

### 貪欲→焼きなましへの発展は慎重に (AHC055 公式解説)

「貪欲で動く評価関数をそのまま SA に使う」は **悪手** になりやすい:
- 序盤効率的でも終盤スコアとの相関が低い評価関数では SA が機能しない
- SA 用の評価関数を別途設計するか、別のアプローチを検討する

## 6. 谷が深い問題への対処 (AHC058 公式解説)

通常の SA では「集中投資先を変える」等の大きな変更が受け入れられない場合:

**対策 1: チェックポイント方式の初期解**

多段階貪欲で強い初期解を作り、SA は微調整に使う:

```
step1: チェックポイント1 (中間目標) への到達を最速化する貪欲
step2: チェックポイント2 への到達を最速化する貪欲
step3: 最終目標に特化した貪欲
```

**対策 2: 巨大近傍**

1 回の遷移で複数ターン・複数要素をまとめて変更する近傍を追加する。

## 7. 高速化

### 差分計算

全スコアを毎回再計算するのでなく、近傍変更分だけ更新する。スコアが O(N) の場合、差分計算で O(1) または O(変更サイズ) にできることが多い。

### 前向き・後ろ向き確率の保持 (AHC051)

スコアが経路上の確率の積の場合:
- `f[i]` = 初期ノード→ノード i の確率
- `b[q][i]` = ノード i →処理器 q の確率

を保持することで、辺 1 本の変更を O(1) で評価できる。

### 交差判定の高速化 (AHC051)

- **事前列挙**: 全辺ペアの交差を事前計算して管理
- **AABB 木**: 動的な交差判定に使う

## 8. デバッグ・チューニングの流れ

1. **まず時間をかけた解を観察する** (AHC051 公式解説の Tips)
   - どのような解が出ているか把握
   - 明らかに悪いパターンがあれば近傍で対処できないか検討

2. **採択率を確認する**
   - 序盤に採択率が低い → T_start を上げる
   - 終盤も採択率が高い → T_end を下げる

3. **イテレーション数を確認する**
   - 少なすぎる (< 10^5) → 差分計算・近傍の計算量削減が必要
   - 1 秒で 10^6 回程度が目安

4. **T_start / T_end のチューニング**
   - `cerr` で採択率を出力しながら調整
   - または Optuna で自動チューニング
